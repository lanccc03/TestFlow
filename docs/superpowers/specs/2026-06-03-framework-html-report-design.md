# Framework HTML Report Design

Date: 2026-06-03

Status: Approved for implementation planning

## Context

The current report detail flow is generated by TestFlow. When an execution
finishes, `save_execution_report()` writes `testflow-report.json` under
`data/reports/<task_id>/` and the frontend report detail page renders the stored
task, steps, logs, and attachments.

The automation framework already generates its own HTML report. That report is
the source of truth for report detail. It is generated inside the framework's
own report directory, not under TestFlow's `data/` directory. TestFlow should
not copy, regenerate, or reinterpret the framework report as a platform report.

## Decision

Prefer the framework-generated HTML report whenever the framework provides one.
TestFlow will persist only a reference to the framework report and keep the
existing structured task detail as a fallback for executions that do not yet
produce a framework report.

The framework report reference is a first-class report field, not a regular
attachment. This keeps screenshots, traces, and logs separate from the report
entry point and avoids filename guessing.

## Report Reference

Add a framework report reference model to execution reports:

- `kind`: currently `html`.
- `title`: display name, defaulting to `框架报告`.
- `source`: `file` or `url`.
- `entry`: for `file`, the HTML entry file path; for `url`, the external URL.
- `root_dir`: for `file`, the report directory that contains the HTML and
  related assets.

For local file reports, the framework runtime reports both the root directory
and the entry HTML file. TestFlow stores those paths as metadata. It does not
move the files into `data/reports`.

## Adapter Contract

Extend the framework event contract so the runtime can return the HTML report
entry when it finishes. The preferred shape is a dedicated event type:

```text
framework_report
  task_id
  report_kind = "html"
  report_root_dir
  report_entry
  report_title
```

The event can be emitted near the end of the run, before `run_finished`. The
runner records the reference on the task/report state and continues normal task
status handling.

The mock runtime can emit no framework report by default so existing fallback
behavior remains testable. A focused test runtime can emit a fake HTML report to
cover the new path.

## API Behavior

`GET /api/reports/{task_id}` continues to return the execution task snapshot,
attachments, and fallback data, and adds `framework_report` when available.

For local HTML reports, add a serving endpoint:

```text
GET /api/reports/{task_id}/framework-report
GET /api/reports/{task_id}/framework-report/{asset_path}
```

The first endpoint returns the entry HTML. Asset requests resolve under the
stored `root_dir` so relative CSS, JavaScript, images, and other report assets
can load.

For URL reports, the API can return the URL directly and the frontend opens it
without proxying.

## Path Safety

Because framework reports live outside TestFlow's data directory, the backend
must validate every local report path before serving it:

- Resolve `root_dir`, `entry`, and requested assets to absolute paths.
- Reject missing report references.
- Reject paths that escape `root_dir`.
- Reject directories as file responses.
- Return `404` for missing files and `403` for path traversal attempts.

This allows external framework report directories without exposing arbitrary
local files.

## Frontend Behavior

The report detail page uses this priority:

1. If `framework_report.kind === "html"`, show the framework report entry first.
2. For local file reports, render the backend framework-report URL in an iframe
   and provide an open-in-new-window action.
3. For URL reports, provide an open action to the external URL.
4. If no framework report exists, render the existing TestFlow structured
   summary, steps, logs, and attachments.

The existing structured detail remains useful for mock runs, failed adapter
integration, and executions created before this feature.

## Persistence

TestFlow keeps the execution history tables and task snapshot because they are
needed for lists, status filters, task restoration, and fallback details.

Stop writing `testflow-report.json` as the report detail artifact. The database
stores the task snapshot, attachment index, and optional framework report
reference. The framework's HTML files remain in the framework report directory.

## Error Handling

If the framework report reference is invalid or the file disappears, the report
detail page should show the regular TestFlow fallback detail and a concise
message that the framework HTML report is unavailable.

If the task ends with `error` before the framework can produce a report, no
framework report reference is required. The fallback detail remains the source
for the platform error message.

## Testing

Backend tests should cover:

- Persisting and reading a report with a framework HTML report reference.
- Serving the local HTML entry file from an external framework directory.
- Serving relative report assets from the same root directory.
- Rejecting path traversal outside the report root.
- Returning fallback report details when no framework report exists.
- No longer writing `testflow-report.json`.

Frontend tests should cover:

- Report detail prefers the framework HTML report when present.
- Report detail falls back to structured steps and logs when no framework
  report exists.
- Missing framework report errors do not hide fallback task details.
